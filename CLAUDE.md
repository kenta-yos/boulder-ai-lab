## WHY：このプロジェクトは何か

BoulderAILab は、撮影済みのボルダリング動画をアップロードすると、AIが「なぜ落ちたのか」「どうすれば成功するか」を具体的に返し、その結果が自分の傾向として蓄積されていく個人用アプリ。非商用。オーナー本人に加え、友人が使う可能性も想定する。

思想：単なる「動画採点機」ではなく、継続的なコーチとの関係を作る道具。1本ごとの解析→チャットで深掘り→記録が蓄積→傾向にもとづくトレーニング提案、という流れ。

コアUXループ：登る → 動画をアップ → 休憩中に結果を見る → 次トライに活かす。「1トライ→数分休憩」という競技特性に最適化する。

## WHAT：技術スタック

- フロント / オーケストレーション：Next.js（Vercelにデプロイ）
- 言語：TypeScript
- データベース：Neon（PostgreSQL 18）。ORMは Prisma。Neonプロジェクトは作成済み（Project ID: `damp-hall-14764204`、Project name: `boulder-ai-lab`、Region: AWS Asia Pacific 1 Singapore、Free plan、Neon Authはオフ）。
- ベクトル検索（RAG用）：Neonの pgvector 拡張。DBはNeon1つで賄う。
- 動画ファイル保管：Blobストレージ（Vercel Blob）
- フレーム抽出：クライアント側（ブラウザ）で実行。動画をサーバー関数に直接送らない。
- LLM：Claude API（統合・コーチング担当）。将来 Gemini を動画読みに併用（差し替え可能に設計）。
- 重い画像認識（ホールド検出・姿勢推定）：将来フェーズで外部GPU（Modal / Replicate）に逃がす。

### 重要な制約
- Vercelのサーバー関数にはリクエストサイズ上限（約4.5MB）と実行時間制限があるため、動画そのものを関数に送らない。動画はBlobへ直アップし、フレーム抽出はブラウザ側で行う。
- LLM呼び出しは `analyze(input) → feedback` という1枚のインターフェースにまとめ、実装（Claude / 将来のGemini）を差し替え可能にする。

## HOW：アーキテクチャ概要

処理パイプライン（完成形）：
1. 動画アップ（Blobへ）＋ ブラウザでキーフレーム抽出
2. 知覚レイヤー（将来）：ホールド検出・3D姿勢推定・数値計算（角度/腰と壁の距離/重心/接地イベント）
3. 動画読み（将来）：Geminiが生動画で動的ムーブ・タイミングを言語化
4. RAG：弱点に応じてNeon(pgvector)から関連知識を取得
5. 統合：Claudeが「数値＋動画メモ＋RAG知識」を受け取り、日本語コーチング（敗因・処方・数値裏付き指摘・タイムスタンプ）を生成
6. 保存：解析結果・スコア・指摘をDBに記録

原則：Claudeには「感想」ではなく「数値」を渡してグラウンディングする。測定に裏打ちされた指摘にするため。

## データモデル（要点）

詳細な定義とER図は `SPEC.md` の「データモデル」章を参照。主要エンティティ：

- マスタ：`User` / `Gym` / `Grade`(ジムごと) / `Skill`(技術軸=採点8軸) / `Technique`(技術知識=RAG) / `Drill` / `Style`(課題特性)
- 解析：`Analysis`(1動画1件) / `Problem`(任意・軽量)
- 解析の子：`AnalysisScore`(解析×技術軸) / `Finding`(指摘＋秒数＋サムネ) / `ChatMessage`
- 提案：`TrainingSuggestion`

決定事項：セッションは持たない（日付＋ジムで擬似的に束ねる）。課題(`Problem`)は任意・軽量で残す。`Style`タグは基本、解析ごとに自動付与。

## コーディング規約

- ES Modules（`import`）を使う。`require()` は使わない。
- 名前付きエクスポートを優先する。
- 環境変数（APIキー、DB接続文字列など）はコードに直書きせず `.env.local` に置き、`.gitignore` で除外する。秘密情報を絶対にコミットしない。
- `.env.local` はオーナーに手作業で作らせず、Claude Code が作成・設定を主導する。オーナーが持っている値（Neonの接続文字列、Claude/その他のAPIキー）を尋ね、正しい変数名で `.env.local` に書き込み、同時に `.gitignore` に `.env.local` が入っていることを必ず確認する。設定後、値が正しく読めているかの確認方法もオーナーに教える。
- Neonへの接続には、Neonダッシュボードの接続文字列を `.env.local` の `DATABASE_URL`（および Prisma が要求する場合は `DIRECT_URL`）に設定する。Neonはpgvectorをサポートするので、RAG用のベクトルも同じDBで扱う。
- DBスキーマ変更は Prisma のマイグレーションで行う。
- 秘密情報（APIキー、動画URL）をログやURLパラメータに含めない。
- 新しいライブラリを追加するときは、理由を一言説明してから追加する。
