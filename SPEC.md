# SPEC.md — BoulderAILab 詳細仕様

アプリ名：**BoulderAILab**（自分の登りをAIで解析・蓄積する、ボルダリングのコーチ兼アナリスト）。個人用（非商用）。このファイルは詳細仕様の参照ドキュメントです。

GitHubリポジトリ：https://github.com/kenta-yos/boulder-ai-lab作業の前提ルールと要点は `CLAUDE.md` を参照してください。

---

## 1. 概要

### 1.1 目的
BoulderAILab は、撮影済みのボルダリング動画をアップロードすると、AIが「なぜ落ちたのか」「どうすれば成功するか」を具体的に返し、その結果が自分の傾向として蓄積されていくアプリ。基本は自分用だが、友人が使う可能性も想定する。

### 1.2 思想
「動画採点機」ではなく、継続的なコーチとの関係を作る道具とする。1本ごとの解析は入口であり、そこからチャットで深掘りし、記録が蓄積され、傾向に基づく提案へつながる。単発で完結しつつ、使うほど「相棒」になっていく設計。

### 1.3 コアUXループ
```
登る → 動画をアップ → 休憩中に結果（一言＋チャット）→ 次トライに活かす
                          ↓（自動）
                    記録に蓄積 → 傾向が見える → トレーニング提案
```

## 2. 画面仕様

### 2.1 ① アップロード画面
- 撮影済み動画をアップロードする方式を中心とする。
- アップ時に課題情報を付与：ジム、グレード、（任意で）課題、（任意で）落下箇所。
- グレードを最初に取ることが重要。コーチングAIが「そのグレードの理想値」で採点する基準になる。
- アップと同時に、裏でフレーム抽出（＋将来は知覚レイヤー）が走り始める。

### 2.2 ② 解析結果画面（アプリの顔）
縦一枚で「一言 → 根拠 → 深掘り」の階層。
1. 一言サマリー：敗因（なぜ落ちたか）と処方（どうすれば成功するか）の2点のみを大きく。休憩中に数秒で読める密度。
2. 根拠（数値裏付き）：「腰が壁から18cm（このグレードの理想は10cm以内）」のように、測定に裏打ちされた指摘を1〜2点。（Phase 1以降）
3. チャット深掘り：解析結果を文脈に持ったまま対話で掘れる。各指摘にはタイムスタンプが紐づき、AIが「クルクスで腰が離れている」と言うと、その瞬間のサムネが横に出る。サムネをタップすると動画がその秒数へジャンプする。（サムネ連動はPhase 1以降）

### 2.3 ③ 記録・傾向画面
- 1本ごとの解析が自動で蓄積される。
- 技術軸ごとのスコアを時系列チャートで表示（推移が見える）。
- フィルタ：ジム別 / グレード別 / 課題特性別。

### 2.4 ④ トレーニング提案画面
- ③で見えた弱点の傾向から、傾向ベースでドリルを提案する（単発の指摘ではなく、蓄積された傾向にもとづく）。

## 3. LLM・処理アーキテクチャ

性質の異なる仕事を、それぞれ最も得意な手段に分業させる。

### 3.1 知覚レイヤー（非LLM・Phase 1以降）
硬い数値をここで計算で出す。
- ホールド検出：クライミング用にfine-tuneしたYOLO ＋ SAM。
- 骨格推定：RTMPose / MediaPipe で2D → MotionBERT / SAM 3D Body で3D化。
- 数値計算：股関節角度、腰と壁の距離の時系列、重心の軌跡、接地イベント。
- 最難所：ホールド検出と、遮蔽（体で隠れる手足）下での正確な3D姿勢推定。

### 3.2 動画読みレイヤー ＝ Gemini（Phase 2以降）
- 生のクリップをネイティブに理解し、動きの質感とタイミング（ためらい、体幹の締まり、動的ムーブのリリースの早さ）を言語化する。

### 3.3 統合・コーチングの脳 ＝ Claude
- 知覚レイヤーの数値 ＋ 動画読みメモ ＋ RAG知識を統合し、優先順位をつけ、グレードに合わせ、怪我に配慮し、日本語ネイティブでコーチングに翻訳する。

### 3.4 下働き ＝ 安価なモデル（Claude Haiku / Gemini Flash-Lite）
- フレーム選定、タグ付け、構造化抽出、ドリル引き当てなどの雑用。

### 3.5 RAGの埋め込み
- `Technique`（技術知識）と日本人プロのTipsをベクトル化し、Neonのpgvectorに格納。弱点に応じて関連チャンクを引く。

### 3.6 最重要原則：グラウンディング
Claudeには「感想」でなく「数値」を渡す。測定に裏打ちされた具体的なコーチングにするため。

### 3.7 差し替え可能設計
`analyze(input) → feedback` のインターフェースを1枚かませ、Claude実装（静止画）と将来のGemini実装（動画）を切り替え可能にする。UI・データ層・プロンプトは共通。

## 4. 技術スタックと制約

- フロント/オーケストレーション：Next.js（Vercel）
- DB：Neon（PostgreSQL ＋ pgvector）、Prisma
- 動画：Blobストレージ（Vercel Blob）、署名付きURLで直アップ
- フレーム抽出：クライアント側（`<video>`→canvas、または ffmpeg.wasm）
- GPU処理（Phase 1以降）：Modal / Replicate

制約：
- 動画をVercel関数に直接POSTしない（ボディ上限 約4.5MB、実行時間制限、永続FSなし）。→ Blob直アップ＋クライアント側フレーム抽出。

## 5. データモデル

### 5.1 決定事項
- セッションは持たない（日付＋ジムで擬似的に束ねる）。
- 課題(`Problem`)は残すが任意・軽量（同じ課題への複数トライの進捗を追うため）。
- 技術軸(`Skill`)＝自分の能力の分解、課題特性(`Style`)＝壁側の性質の分解。得意/不得意は両者の交差点に現れる。
- `Style`タグは基本、解析ごとに自動付与。課題を定義した場合はそちらに寄せる。

### 5.2 エンティティ
マスタ層
- `User`：id, name, homeGymId
- `Gym`：id, name, area
- `Grade`：id, gymId, label, orderIndex（※ジムごとに基準が違うためGymに紐づく）
- `Skill`：id, name, category（採点軸＝チャート・指摘の共通言語）
- `Technique`：id, skillId, name, description（第2層・RAG知識）
- `Drill`：id, skillId, name, instructions
- `Style`：id, category, label（課題特性タグ）

解析層
- `Problem`：id, gymId, gradeId, name（任意・軽量）
- `Analysis`：id, userId, gymId, gradeId, problemId?, videoUrl, summary, prescription, perception(JSON), createdAt

解析の子
- `AnalysisScore`：id, analysisId, skillId, score, evidence（③チャートの元データ）
- `Finding`：id, analysisId, skillId, tSec, frameUrl, comment（②サムネ連動の元）
- `ChatMessage`：id, analysisId, role, content, refTSec

提案層
- `TrainingSuggestion`：id, userId, skillId, drillId, rationale

### 5.3 主な関係
- Gym 1–* Grade / Problem
- User 1–* Analysis / TrainingSuggestion
- Analysis 1–* AnalysisScore / Finding / ChatMessage
- Skill 1–* AnalysisScore / Finding / Technique / Drill / TrainingSuggestion
- Drill 1–* TrainingSuggestion
- Problem *–* Style（1課題が複数特性）
- Style *–* Skill（課題特性が要求する技術軸）


## 6. 技術体系（二層構造）

出典の骨格は The Climbing Bible（ノルウェー代表コーチ陣の教本）の技術分類（フットワーク／グリップポジション／バランス／テンションと力の方向／ダイナミクス）に、日本語のクライミング用語を突き合わせて構成した。

### 6.1 第1層：採点する技術軸（`Skill`＝8軸）
1. フットワーク（足の置き精度・静かな足・エッジ/スメアの使い分け・足荷重）
2. ボディテンション（支持）（体幹で体を壁に留める・足が切れない）
3. 重心・バランス（重心を足に乗せる・回転＝バーンドアを抑える）
4. 腰の位置（壁との距離。知覚レイヤーが数値で直接測れる軸）
5. 保持（ホールディング）（ホールドに応じた持ち方の適切さ）
6. 力の方向（ホールドに対し正しい方向へ・対抗を効かせる）
7. ムーブ効率・省エネ（適切なムーブ選択・腕を伸ばす・レスト・無駄のなさ）
8. ダイナミクス（動的ムーブ・デッドポイントのタイミング・静と動の使い分け）

採点しないコーチング話題（チャートに乗せず②のチャットで扱う）：オブザベ/ルートリーディング（戦術）、メンタル（怖さ・コミット）。

### 6.2 第2層：知識としての技術（`Technique`＝RAGハンドブック）
- フットワーク下：エッジング、スメア、トウフック、ヒールフック、手に足、足の入れ替え
- バランス下：ダイアゴナル（正対/側対）、インサイド/アウトサイドフラッギング、カウンターバランス
- ムーブ効率下：ドロップニー（キョン）、ステミング、レイバック、マントリング、ニーバー、クロス、バンプ、マッチ
- ダイナミクス下：デッドポイント、ランジ、ダイノ、ダブルダイノ、サイファー
- 保持下：クリンプ/ハーフクリンプ、オープンハンド、ピンチ、スローパー、ポケット、ガバ

### 6.3 課題特性（`Style`）の切り口
複数タグの組み合わせで表現。知覚レイヤー＋Geminiで自動タグ付け可能。
- 壁角度：スラブ / 垂壁 / 強傾斜（オーバーハング）/ ルーフ
- 主要ホールドタイプ：ガバ / クリンプ（カチ）/ スローパー / ピンチ / ポケット
- 課題の性質：パワー系 / 保持系 / バランス系 / ランジ・動的系 / 足自由 / コーディネーション系 / 傾斜デリケート系
- 要求される主要技術：`Skill`（8軸）への橋渡し

## 7. コスト・プライバシー

### 7.1 コスト感（個人趣味用）
- 完璧構成で1解析あたり おおよそ $0.25〜0.50（主なドライバーは知覚のGPUと統合LLM）。
- Phase 0（Claude単体・静止画）なら1解析あたり $0.02〜0.06 程度。
- 毎日数本アップしても月数百〜数千円レンジの想定。桁感として扱う。

### 7.2 プライバシー
- 人体が映る動画を扱うため配慮する。
- Claudeの有料APIはデフォルトで学習に使われない。
- Geminiの無料枠はコンテンツが製品改善に使われるため、動画を流す場合は有料ティアを使う。

## 8. 段階的ロードマップ（受け入れ基準つき）

1フェーズが動作確認できてから次に進む。

### Phase 0：最小で動かす
含む：動画アップロード（ローカル選択）、ブラウザでのフレーム抽出、Claudeで敗因＋処方の生成と表示、解析結果を文脈にした深掘りチャット、解析結果のDB保存と一覧表示。UIの4画面の骨格。
受け入れ基準：短い動画を選ぶと、数十秒以内に「敗因」と「処方」がテキストで表示され、その内容についてチャットで質問でき、結果が記録一覧に残る。

### Phase 1：グラウンディング
含む：知覚レイヤー（ホールド検出・3D姿勢・数値計算）をGPUサービスで実装、数値をClaudeに渡すグラウンディング、指摘へのタイムスタンプ付与とサムネ連動。
受け入れ基準：指摘が数値（例：腰と壁の距離）に裏打ちされ、各指摘のサムネをタップすると動画が該当秒へジャンプする。

### Phase 2：動画読みと蓄積
含む：Geminiの動画読み（動的ムーブ）、`Style`自動タグ付け、技術軸別＋課題特性別の傾向チャート、傾向ベースのトレーニング提案。
受け入れ基準：解析を重ねると技術軸ごとのスコア推移がチャートで見え、課題特性別にフィルタでき、弱点に応じたドリルが提案される。

### Phase 3：ハンドブック拡充
含む：`Technique`のRAGコーパスと日本人プロTipsの整備、深掘りチャットでの知識参照。
受け入れ基準：チャットでの深掘りが、整備された技術知識に基づいた回答を返す。

## 9. 未確定・要検討事項

- 採点8軸のうち「保持」「力の方向」は数値化が難しい。暫定：LLMの定性判断で採点しつつ信頼度を低めに明示。運用しながら「採点せずコーチング話題に回す」への切替も検討。
- `Skill` / `Style` の初期マスタ投入方法（手動シード／自動生成の範囲）。
- `Grade`マスタの初期投入（自分が使うジムだけ手登録し、育てる前提）。
- Gemini / Claude の具体モデルID・バージョンは実装時点の最新を確認して確定。
